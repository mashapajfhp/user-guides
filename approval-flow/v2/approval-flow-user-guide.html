What is Approval Flow?

Overview

Approval Flow defines who reviews and approves employee requests—leaves, expenses, payroll transactions, tickets—and in what order. Configure simple or complex approval hierarchies with role-based routing, sequential or parallel approvers, and conditional logic based on request attributes.

Create standard flows (company-wide rules) or advanced flows (conditional routing based on amount, policy type, department, or 60+ data points). Manage flows across 14+ request types including leaves, air tickets, loans, expenses, payroll transactions, attendance regularization, and employee violations.

Key Benefits

  • Eliminate manual approval tracking with automated routing
  • Enforce maker-checker controls for financial transactions
  • Configure multi-level hierarchies matching organizational DOA requirements
  • Reduce approval delays with parallel approver options and backup designations
  • Route requests differently based on amount, type, department, or custom criteria
  • Maintain audit trails showing who approved what and when

Product Foundation Overview

How Approval Flow Fits

Approval Flow is a configuration system that defines routing rules for requests requiring approval. It acts as a policy layer that sits between employee submissions and final processing.

Mental model: Approval Flow (routing rules) → Applied to request type → Routes to designated approvers → Approved/rejected → Request processed or returned

One approval flow configuration can control routing for hundreds of requests across multiple modules (leaves, expenses, payroll transactions).

Key Decisions Before Setup

Answer these questions before configuring Approval Flow:

  • Who needs to approve what? — Identify which roles should approve which request types (Line Manager, HR, Finance)
  • Sequential or parallel approval? — Decide if approvals must happen in order or if any approver can act
  • How many approval levels? — Determine if single-step or multi-level approval is needed
  • What happens when an approver is unavailable? — Plan for backup approvers or delegation rules

Initial Setup & Configuration

Configuration Walkthrough

Approval flows are configured per module (Leave, Air Ticket, Loan, Expense, etc.) at Settings → Approval flows. Each module has its own tab where you define who approves requests and in what order.

Settings → Approval flows

Each tab displays existing approval flows for that module. Standard flows apply company-wide; advanced flows use conditional routing based on criteria.

Standard Approval Flow Setup

Standard flows define a single approval hierarchy for all requests in a module. This is the most common configuration for organizations without complex routing needs.

Step 1: Navigate to Module Tab

Select the module tab (Leave, Air Ticket, Loan, etc.) where you want to configure approvals.

Step 2: Add Approval Steps

Click the "Add approval step" or similar button to define the approval hierarchy.

Step 3: Select Approver Roles

For each step, choose which roles can approve:

  • Line Manager — Direct manager of the employee (1 level, 2 levels, etc.)
  • System Roles — Super Admin, Time Off Manager, Expense Manager, etc.
  • Custom Roles — Department Manager, Office Manager, Sales Director, CEO, etc.

Step 4: Configure Approval Logic

Define how approvers in the same step interact:

  • AND Condition — All approvers in the step must approve
  • OR Condition — Any one approver in the step can approve
  • Sequential Steps — Step 1 must approve before Step 2 sees the request

Step 5: Add Multiple Approvers (Optional)

For redundancy, add multiple approvers per step. If one is unavailable, another can approve.

Step 6: Save Configuration

Save the approval flow. It applies immediately to all new requests in that module.

Advanced Approval Flow Setup

Advanced flows route requests based on conditions (amount, policy type, department, office, etc.). Multiple advanced flows can exist per module, each with its own criteria and priority.

Step 1: Create Advanced Flow

Click "+ Add advanced flow" on the module tab (e.g., Air Ticket, Leave).

Step 2: Name and Describe

  • Flow name — Descriptive name (e.g., "High-value air ticket approval")
  • Description — Brief explanation of when this flow applies

Step 3: Add Criteria

Click "Add criteria" to define conditions that trigger this flow.

Select from 60+ data points:

  • Policy Allowance Amount → Amount — Route based on monetary value
  • Policy Allowance Amount → Currency — Route based on currency
  • Employee → Department — Route based on department
  • Employee → Office — Route based on office location
  • Leave Policy → Policy Name — Route based on leave type

Choose an operation:

  • is equal to
  • is not equal to
  • is greater than
  • is less than
  • is between
  • is not between
  • contains
  • does not contain

Enter the value to match against (e.g., 3000 for amount, "Sales" for department).

Step 4: Configure Criteria Logic

If multiple criteria exist, choose:

  • Execute when ALL criteria(s) match — All conditions must be true
  • Execute when ANY criteria(s) match — Any one condition triggers the flow

Step 5: Add Approval Steps

Define the approval hierarchy for requests matching this flow's criteria.

Select approver roles (same options as standard flows: Line Manager, system roles, custom roles).

Step 6: Set Priority

If multiple advanced flows match a request, the highest priority (lowest number) flow applies. Use the ellipsis menu to reorder flows.

Step 7: Save Advanced Flow

Save the flow. It applies immediately to new requests matching the criteria.

Common Configuration Patterns

Pattern 1: Simple Line Manager Approval

All requests go to direct manager (1 level).

  • Use case: Small organizations with flat hierarchy
  • Setup: Single step, Line Manager (1 level)

Pattern 2: Two-Level Sequential Approval

Direct manager approves first, then HR manager.

  • Use case: Organizations requiring HR oversight
  • Setup: Step 1 - Line Manager (1 level), Step 2 - Time Off Manager

Pattern 3: Amount-Based Routing

High-value requests require additional approval.

  • Use case: Financial controls for expenses/air tickets
  • Setup: Advanced flow with criteria "Amount > 3000" → Finance Manager + Line Manager

Pattern 4: Department-Specific Approval

Different departments have different approval chains.

  • Use case: Large organizations with decentralized approval
  • Setup: Multiple advanced flows, each with criteria "Department = [name]" → Department Manager

Pattern 5: Ultimate Approver for Urgent Cases

CEO or HR Director can bypass normal approval chain.

  • Use case: Emergency approvals when normal approvers unavailable
  • Setup: OR condition with Ultimate Leave Approver role + normal approval chain

Granular visibility controls that align with custom multi-level approval workflows are not available.

Approval Flow Configuration Rules

  • Default Processing: If no approval flow is configured, transactions process automatically without approval.
  • Rejection Requirements: Rejection requires remarks for amendment guidance.
  • Multi-Level Sequence: Multi-level approvals must complete in sequence.
  • Email Notifications: Email notifications are required for pending approvals.
  • Approver Hierarchy: Approvers must be in the approval hierarchy to act on requests.

Edge Cases

  • Approver Unavailability: When approver is on vacation, illness, or extended leave, multiple approvers per step provide redundancy.
  • Circular Dependencies: Avoid circular approval dependencies in configuration.
  • Transaction Edits: Transaction edited after submission but before approval requires resubmission.
  • Role Removal: Approver role removed mid-workflow requires workflow reconfiguration.
  • Multiple Pending Transactions: Multiple transactions pending for same approver queue in sequence.
  • Partial Approval Rejection: Rejection after partial multi-level approval restarts cycle.
  • Timeout Handling: No automatic timeout for pending approvals; manual follow-up required.
  • Self-Approval: Approver trying to approve their own transaction is blocked.

FAQs & Support

Frequently Asked Questions

Who can configure approval flows?

Only Super Admin users can configure approval flows. Line Managers and other approvers must first be assigned their roles in Role Management.

What happens if one approver in a step rejects a request?

If any approver in a step rejects the request, the entire request is immediately rejected and cannot be moved back to pending status.

Can I set up parallel approvals using OR conditions?

Yes. Approvers can be configured to approve in parallel using OR conditions, where approval from any one approver in the group is sufficient to proceed.

How do I ensure approval requests route correctly?

Each role must be properly assigned to relevant employees in Role Management. Without proper role assignments, approval routing will fail.

Can I add multiple approval levels?

Yes. You can add multiple approval steps in sequence, with each step containing one or more approvers who must approve before the request moves to the next level.

What's the difference between AND, OR, and Step conditions?

AND requires all approvers to approve, OR requires any one approver to approve, and Step conditions create sequential approval levels where each level must complete before the next begins.

Can approvers delegate their approval authority?

Yes. Approvers can delegate approval authority if they are unavailable, ensuring requests continue to flow even when primary approvers are out of office.

What types of requests support approval flows?

Approval flows are available for leaves, air tickets, loans, expenses, payroll transactions, leave salary requests, attendance regularization, leave encashment, accounts payable, employee violations, shift change requests, business trips, grievances, and hourly permissions.
