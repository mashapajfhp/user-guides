What is Air Tickets?

Overview

Air Tickets manages employee air ticket encashment requests through a structured approval workflow. Employees submit requests for ticket reimbursement or encashment, which route through configured approvers and automatically sync to payroll when approved.

Admins review requests, track status across pending/approved/processed/rejected tabs, and process encashments to payroll. Employees submit requests and monitor approval status.

Key Benefits

  • Eliminate manual tracking of air ticket requests through spreadsheets or email chains
  • Standardize encashment request submission with required fields and attachments
  • Automate approval routing to line managers and configured approvers
  • Sync approved encashments directly to payroll without manual data entry
  • Track request history and status in real-time with activity feeds
  • Enable employees to self-serve ticket requests from their profile

Strategic Context

  • Drives payroll deductions or reimbursements based on ticket policy
  • Controls which employees are eligible for encashment requests

How It Connects

Transactional feature that creates payroll adjustments after approval routing completes.

Product Foundation Overview

How Air Tickets Fits

Air Tickets is a transactional request system within the Employee Tickets framework, processing employee air ticket encashment and reimbursement requests. It routes requests through approval flows and creates payroll adjustments when approved.

Mental model: Employee submits air ticket request → Routed to approver(s) → Approved → Creates payroll adjustment → Processed in payroll run

Impacts payroll processing for all employees with approved air ticket requests in the current period.

Key Decisions Before Setup

Answer these questions before configuring Air Tickets:

  • Which ticket types do you need? — Determine if you need reimbursement requests, encashment requests, or both
  • Who should approve requests? — Define approval routing (Line Manager, HR, Finance, or multi-level)
  • What information must employees provide? — Decide required fields (ticket cost, travel dates, destination, attachments)
  • How will approved requests flow to payroll? — Plan whether to use manual processing or automated workflow creation

Related Features

These Bayzat features work together with Air Tickets:

  • Employee Tickets — Parent module that provides the ticket creation and approval framework
  • Approval Flows — Routes air ticket requests to configured approvers based on rules
  • Payroll — Receives approved air ticket amounts as pay adjustments for processing
  • Workflows — Automates encashment request creation based on triggers (see Zendesk article on Auto Encashments)
  • Employee Profile — Provides employee data and reportee relationships for request routing

Initial Setup

First-Time Configuration

Before employees can submit air ticket requests, configure the ticket type and approval routing.

Step 1: Configure Air Ticket Type

Navigate to the ticket type settings to enable air ticket requests for your organization.

  1. Navigate to Settings → Tickets
  2. Locate the Air Tickets ticket type (premade by system)
  3. Configure employee mapping: select which employees are eligible to submit air ticket requests
  4. Assign admin roles: define which roles can view and manage air ticket requests
  5. Save configuration

Note: Air Tickets is a premade ticket type. System fields are view-only and cannot be modified. You can edit the name, description, employee mapping, and admin roles.

Step 2: Set Up Approval Flow

Define who approves air ticket requests and in what order.

  1. Navigate to Settings → Approval Flow
  2. Locate Air Tickets in the ticket types list
  3. Configure the default approval flow (typically Line Manager)
  4. Set up advanced flows if needed (conditional routing based on amount, department, or other criteria)
  5. Save approval configuration

Default configuration: Air ticket requests route to the employee's Line Manager for approval. You can modify this to add additional approval steps or conditional routing.

Step 3: Add New Employees to Ticket Type

When onboarding new employees, manually add them to the Air Tickets ticket type if they should have access.

  1. Navigate to Settings → Tickets
  2. Open Air Tickets configuration
  3. Update employee mapping to include new hires
  4. Save changes

Important: New employees are not automatically added to ticket types. Default mapping includes all active employees at the time of initial setup, but new hires must be manually added as needed by your organization.

Optional: Configure Workflow Automation

Set up workflows to automate air ticket encashment request creation.

  1. Navigate to Settings → Workflows
  2. Create new workflow and select "Air Tickets Auto Encashment" recipe
  3. Set trigger time (date and hour when workflow runs)
  4. Define criteria: choose between fully approved requests or requests requiring approvals
  5. Select encashment request type
  6. Assign employee responsible for processing
  7. Save and activate workflow

Note: Workflow time is based on your account location (UAE or KSA). The minute selector is non-functional; only hour-level precision is available.

Known System Limitations

Line Manager Notifications (Medium): The platform cannot notify or tag line managers about ticket processing when they are outside the standard approval flow. No automated notification system exists for proxy requesters or managers raising requests for employees. Status: Pending (TSSD-3954)

Payroll Period Adjustments (Medium): Moving processed air ticket transactions between payroll periods requires manually reopening periods, deleting transactions, and moving them to pending status. No streamlined mechanism exists for cross-period transaction movement. Status: Resolved (TSSD-3366)

Employee Self-Service Interface (Low): Employees lack a unified interface to manage air ticket requests under various policy conditions. Specific scenarios exist where employees cannot fully interact with or understand their air ticket encashment status. Status: Won't Do (OS-695)

Edge Cases

  • Approval flow change mid-process: Pending tickets migrate to new flow immediately; approved tickets remain unaffected and retain original approval chain
  • Line Manager creating on behalf of employee: Allowed when configured in ticket type settings; ticket shows both creator and employee in audit trail
  • Ticket state transitions: Tickets can move between pending, approved, and rejected states seamlessly; state changes trigger notifications to creator and approvers
  • Category in use warning: System shows warning when editing categories associated with active ticket types; deletion blocked until all associations removed
  • Version mismatch after edits: Existing tickets show "older config" label after ticket type configuration changes; historical tickets retain original field structure
  • Eligibility conflicts: Manual include/exclude overrides filter rules (Department, Office, Country, Gender); explicit inclusions take precedence over automated filters
  • Priority default behavior: If priority not specified during ticket creation, system defaults to Medium priority level
  • Notification preferences: Users can enable/disable alerts on mobile or email independently; notification settings apply to all ticket types globally
  • Custom ticket types with custom fields (Release 3): Extensible system supports any request type beyond attendance and leave (e.g., business trips, loans, resignation, office supplies, shift changes)
  • System lists cannot be deactivated: Built-in dropdown lists used across ticket types are protected; custom lists can be deactivated only if not in use

Data Integrity Safeguards

Historical Ticket Preservation: When ticket type configurations change, existing tickets retain their original field structure and values. A version label indicates tickets created under older configurations.

Approval Chain Integrity: Approved tickets are never affected by approval flow changes. Only pending or partially approved tickets migrate to new flows.

Field Deactivation Protection: Deactivated fields remain visible and accessible in existing tickets to preserve historical data. Only new ticket submissions exclude deactivated fields.

FAQs & Support

Frequently Asked Questions

Can line managers be notified about air ticket requests when they're not in the approval flow?

Currently, the platform only notifies users who are part of the configured approval flow. Line managers outside the approval chain do not receive automatic notifications when requests are submitted or processed.

How do I move processed air ticket encashments between payroll periods?

To move processed transactions, you must manually reopen the payroll period, delete the transaction from the processed period, and move it to pending status in the target period. There is no automated transfer mechanism.

Can employees manage air ticket requests under different policy conditions?

The platform currently lacks a unified interface for employees to fully manage air ticket requests across all policy scenarios. Some policy conditions may limit employee visibility or interaction with their encashment status.

How do I set up automatic air ticket encashment requests?

Navigate to Settings → Workflows, create a new workflow, and select the Air Tickets Auto Encashment recipe. Configure the trigger time, set criteria for encashment, choose the request type, and activate the workflow.

What happens to pending air ticket requests when I change the approval flow?

Pending and partially approved tickets migrate to the new approval flow automatically. Tickets that are already fully approved remain unaffected by the approval flow change.

Can I create custom air ticket types beyond the premade options?

Yes, you can create custom ticket types with configurable fields, eligibility rules, and approval workflows under Settings → Tickets. Custom types extend beyond premade options like attendance regularization or leave encashment.

Why can't I access pay item history when payroll table cells are disabled?

This is a known limitation. When payroll table rows are disabled, the history popup for pay items becomes inaccessible, interrupting the workflow for retrieving historical compensation information.

Can new employees automatically access air ticket types?

No, new employees must be manually added to ticket types as needed by your company. The default employee mapping includes all active employees at the time of ticket type creation, but new hires require manual inclusion.
