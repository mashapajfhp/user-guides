What is Daily Wage Calculator?

Overview

Daily Wage Calculator defines how Bayzat calculates daily wages for three payroll services: salary proration, end-of-service leave encashment, and unpaid leave deductions. Configure whether calculations use working days, calendar days, or custom days to align with labor law requirements.

Each service row displays the current formula. Click the edit icon to change the calculation method and see the formula update dynamically.

Key Benefits

  • Align wage calculations with UAE and KSA labor law (30-day standard)
  • Control how weekends and public holidays affect daily rates
  • Apply different calculation methods to different payroll services
  • See formulas update in real-time as you change settings

Product Foundation Overview

How Daily Wage Calculator Fits

Daily Wage Calculator is a configuration service that defines how daily wages are calculated across three payroll scenarios: salary proration, leave encashment, and unpaid leave deductions. It sits in payroll settings and controls the mathematical basis for all per-day salary calculations.

Mental model: Calculation Method (template) → Applied to Payroll Process → Determines Daily Rate → Affects Employee Pay

One calculation method affects all employees in the organization whose pay involves daily wage calculations.

Key Decisions Before Setup

Answer these questions before configuring Daily Wage Calculator:

  • Which calculation base aligns with your labor law? — UAE and KSA typically use 30 custom days; other regions may use calendar days (28-31) or working days (excludes weekends/holidays)
  • Do you need different calculation methods for different scenarios? — You can set separate methods for salary proration, leave encashment, and unpaid leave deductions
  • Should public holidays be excluded from working days? — Determines whether holidays count toward the divisor when calculating daily rates
  • Will you override daily wage calculations in individual leave policies? — Some leave types may need custom calculation methods that differ from the system default

Related Features

These Bayzat features work together with Daily Wage Calculator. Where available, links point to the published user guide.

  • Payroll Management — Parent module that processes daily wage calculations during payroll cycles and applies salary adjustments
  • Leave Encashment — Uses daily wage calculation to determine cash value when employees convert unused leave to pay
  • Leave Policies — Can override daily wage calculation method for specific leave types using the policy-level toggle
  • Leave Management — Applies daily wage calculation when processing unpaid leave requests to generate payroll deductions
  • Public Holiday Calendar — Provides holiday dates that are excluded from working days when that calculation method is selected

Initial Setup & Configuration

Configuration Walkthrough

Daily Wage Calculator is configured through three system-defined services that control how daily wages are calculated across payroll, leave encashment, and unpaid leave deductions.

Settings → Payroll → Daily Wage Calculation

The section displays three fixed service rows that cannot be added to or deleted. Each row shows the service name and current calculation formula. Click the edit pencil icon on any row to configure its calculation method.

Service 1: Salary Proration

Controls how daily wages are calculated when prorating employee salaries for partial months (new hires, terminations, mid-month changes).

Configuration steps:

  1. Click the edit pencil icon on the Salary proration row
  2. Select calculation base from the Month calculation dropdown
  3. Review the formula that updates dynamically based on your selection
  4. Click Save to apply changes

Calculation method options:

  • Working days — Divides salary by actual working days in the month (excludes weekends and public holidays)
  • Calendar days — Divides salary by total calendar days in the month (28-31 days depending on month)
  • Custom days — Divides salary by a fixed number of days you specify (most UAE and KSA companies use 30 days to align with labor law)

Salary components: The formula automatically includes Basic salary + allowances. You cannot customize which salary components are included in the daily wage calculation.

Service 2: EOS Leave Encashment

Controls how daily wages are calculated when employees cash out unused leave during end of service settlement.

Configuration steps:

  1. Click the edit pencil icon on the EOS leave encashment row
  2. Toggle "Overwrite daily wage calculation in policies" ON to use this calculation for all leave types, or OFF to let individual leave policies define their own calculation
  3. Review the table showing 47+ leave types and their current calculation settings
  4. Click individual leave type links to configure type-specific calculations
  5. Click Save to apply changes

⚠️ Overwrite behavior: When the Overwrite toggle is ON, the Daily Wage Calculator setting takes precedence over any calculation method defined in individual leave policies. When OFF, each leave policy's calculation setting is respected.

Service 3: Unpaid Leave Deduction

Controls how daily wages are calculated when deducting salary for unpaid leave days.

Configuration steps:

  1. Click the edit pencil icon on the Unpaid leave deduction row
  2. Toggle "Overwrite daily wage calculation in policies" ON to use this calculation for all unpaid leave types, or OFF to let individual leave policies define their own calculation
  3. Review the table showing 13 leave types and their current calculation settings
  4. Click individual leave type links to configure type-specific calculations
  5. Click Save to apply changes

Note: Some leave policies may show a custom divisor like 30.41 days instead of the standard calculation. This indicates a policy-specific override is active.

Calculation Method Selection Guide

Choose the calculation method that aligns with your company's payroll policy and local labor law requirements.

Method When to Use Example (AED 6,000 salary)
Working days When you want to exclude weekends and public holidays from the calculation. Divisor changes monthly based on actual working days (typically 20-23 days). January with 22 working days: 6,000 / 22 = AED 272.73 per day
Calendar days When you want to include all days in the month. Divisor changes monthly based on calendar length (28-31 days). January with 31 days: 6,000 / 31 = AED 193.55 per day
Custom days (30) Most common in UAE and KSA. Provides consistent daily wage year-round, aligning with labor law requirements. Divisor is fixed at 30 days regardless of month length. Any month: 6,000 / 30 = AED 200 per day

⚠️ Impact on payroll: Your calculation method choice directly affects salary proration for new hires, terminations, unpaid leave deductions, and leave encashment payouts. Changing the method mid-year may create inconsistencies in employee pay calculations.

Activation & Impact

Configuration changes take effect immediately upon saving. No separate activation step is required.

What happens after configuration:

  • Salary proration calculations for new hires and terminations use the selected method
  • Unpaid leave deductions in the next payroll cycle apply the configured daily wage
  • Leave encashment requests calculate payout amounts based on the configured method
  • End of service settlements use the configured calculation for leave encashment

Payroll cycle timing: Changes made before the payroll cycle closes will affect that cycle's calculations. Changes made after the cycle closes will apply to the next cycle.

Task: Understand Daily Wage Calculation Behavior

Learn how different calculation methods affect daily wage rates and payroll outcomes across various scenarios.

Subtask: Compare Calculation Methods

Method Formula Daily Wage Stability Best For
Working Days (Basic + Allowances) / Working days in month Varies by month (20-23 days typically) Organizations wanting to pay only for actual work days
Calendar Days (Basic + Allowances) / Calendar days in month Varies by month (28-31 days) Simple proration without work week considerations
Custom Days (Basic + Allowances) / Fixed number (e.g., 30) Consistent across all months UAE/KSA compliance, consistent leave encashment calculations

Subtask: Understand Public Holiday Impact

Working Days Method: Public holidays are automatically excluded from the working days count. If a month has 22 working days but includes 2 public holidays, the divisor becomes 20 days, increasing the daily wage rate for that month.

Calendar Days and Custom Days Methods: Public holidays do not affect the divisor. Daily wage remains constant regardless of public holidays in the month.

Subtask: Understand Work Week Configuration Impact

Critical: If using the Working Days method, the daily wage calculation depends on each employee's assigned work week configuration. Employees with 7-day work weeks (no weekends) will have higher working day counts and lower daily wage rates than employees with 5-day work weeks.

Example scenario:

  • Employee A: 5-day work week → 22 working days in January → Daily wage = Salary / 22
  • Employee B: 7-day work week → 31 working days in January → Daily wage = Salary / 31
  • Same monthly salary, different daily wage rates due to work week configuration

Task: Troubleshoot Daily Wage Calculation Issues

Resolve common issues and understand system behavior when daily wage calculations produce unexpected results.

Subtask: Verify Calculation Method Selection

  1. Navigate to Settings → Payroll → Daily Wage Calculation
  2. Click edit on Salary proration to confirm the selected method
  3. If using Custom days, verify the number matches your intended divisor (typically 30)
  4. If using Working days, check employee work week configurations in Employee Structures

Subtask: Investigate Proration Discrepancies

Common Issue: Proration amount doesn't match expected calculation. This usually occurs when the working days count differs from expectations due to work week configuration or public holidays.

Resolution steps:

  1. Check the employee's assigned work week in Employee Structures
  2. Verify public holidays configured for the relevant month
  3. Calculate expected working days: Calendar days - Weekends - Public holidays
  4. Confirm the daily wage formula matches the configured method

Subtask: Handle Leave Salary Accrual Fluctuations

Known Behavior: When using the Working Days method, leave salary accrual amounts vary month-to-month because the daily wage rate changes based on the number of working days in each month. This is expected system behavior, not a bug.

If consistent accrual is required:

  • Switch to Custom Days method with a fixed divisor (e.g., 30 days)
  • This provides consistent daily wage rates across all months
  • Aligns with UAE/KSA labor law requirements for leave encashment

Business Rules & Limitations

High-Impact Limitations

⚠️ EOS and Leave Encashment Calculation Limitations

The system cannot dynamically adjust End of Service (EOS) and leave encashment calculations for employees with mid-month departures and unpaid leave periods. EOS is calculated using full monthly salary without considering actual days worked or payment received. Configuration changes in daily wage calculation are not immediately reflected in the EOS Eligibility section.

Jira References: TSSD-3939, TSSD-4033, TSSD-4742

⚠️ Lack of Configurable Salary Proration Methods

The platform applies a single proration approach across all employees. You cannot customize calculation methods per department, per group, or per employee. The global setting (Working days, Calendar days, or Custom days) applies company-wide.

Jira References: TSSD-4303, TSSD-4882, TSSD-4905

⚠️ Time Off and Payroll Module Synchronization Gap

The platform lacks real-time synchronization between Time Off and Payroll modules. When public holidays are added after leave requests are approved, the system does not automatically recalculate unpaid leave deductions. Manual intervention is required to correct calculations.

Jira References: TSSD-2056, TSSD-4464

Medium-Impact Limitations

Issue Impact Jira Reference

Configuration Constraints

Salary Proration Calculation Methods

The system supports three calculation methods, applied globally:

Method How It Works When to Use
Working days Divides monthly salary by the number of working days in the employee's work week configuration for that month When employees have varied work week schedules (5-day, 6-day, 7-day weeks)
Calendar days Divides monthly salary by the actual number of days in the month (28-31) When proration should account for month length variations
Custom days Divides monthly salary by a fixed number you specify (e.g., 30) When you need a consistent divisor regardless of month or work week

By Design: Working days calculation uses the employee's assigned work week configuration. If a work week has all 7 days configured as working days with no weekends, the system will count 31 working days for a 31-day month. This is correct behavior based on the work week setup.

Jira Reference: TSSD-4292

Working Days Configuration Clarity

Note: When you select "Working days" in the Salary Proration dialog, the system does not display a tooltip or help text explaining how the working days count is determined. The Number of days field disappears, and the formula shows "/ Working days" with no explanation of what value is used. The working days count is derived from the employee's work week configuration.

Jira Reference: TSSD-4292

EOS and Leave Encashment Configuration

⚠️ Disconnected Configuration Sections

The EOS leave encashment settings and daily wage calculation settings are in separate accordion sections with no visible connection. The EOS dialog shows 47+ paid leave types with per-policy calculation methods (Calendar days or Working days). The Overwrite toggle controls whether per-policy settings override the global setting. The End of Service eligibility section shows a separate table with only 1 leave type and a Configure button, disconnected from the EOS leave encashment dialog.

Edge Cases

  • Leave salary accrual with working days setting: When using the "Working days" proration method, the daily wage rate changes based on the number of working days in each month. This causes monthly fluctuations in leave salary accrual calculations, leading to inconsistent cumulative totals. (TSSD-3648)
  • Multiple divisor methods in active use: If your Unpaid Leave Deduction dialog shows different divisor methods across leave types (e.g., some at 30 days, others at 30.41 days, one at calendar days), this is a configuration inconsistency. Standardize divisor methods where possible to avoid confusion.
  • Mid-month departures with partial leave balances: The platform lacks flexible leave accrual handling for mid-month departures. The system defaults to monthly accrual that does not prorate or include partial month leave balances. Manual intervention is required for these edge cases. (TSSD-4742)
  • Work week with no weekends: If an employee's work week configuration has all 7 days set as working days, the "Working days" proration method will count every day in the month as a working day. This is by design but may not match your intended calculation. Review work week configurations to ensure weekends are properly excluded.

Known Workarounds

Working Days Calculation Clarification: When the "Working days" proration method shows an unexpected divisor, verify the employee's work week configuration. The system counts working days based on the assigned work week, not a standard 5-day or 6-day assumption. If the work week has 7 working days configured, the calculation is correct per the configuration. (TSSD-4292 - By Design)

FAQs & Support

Frequently Asked Questions

Why don't my EOS leave encashment settings appear in the End of Service eligibility section?

The EOS leave encashment dialog and End of Service eligibility section are separate configuration areas with no visible connection. Changes in one do not automatically reflect in the other. Configure both sections independently to ensure accurate calculations.

Can I set different salary proration methods for different departments?

No. The salary proration setting applies globally to all employees. You can choose Working days, Calendar days, or Custom days, but only one method applies company-wide.

Why do different leave policies show different daily wage divisors?

Each leave policy can have its own calculation method. Most use 30 days, but some use 30.41 days, calendar days, or custom values. Review the Unpaid Leave Deduction dialog to see which divisor applies to each policy.

How does the system calculate daily pay when working days is selected?

The system divides monthly salary by the number of working days configured in the employee's work week, excluding weekends and public holidays. The exact working days count comes from the employee's assigned work timing schedule.

Will payroll recalculate automatically if I add a public holiday after leave requests are approved?

No. The system does not retroactively recalculate payroll when public holidays are added after leave approvals. Approved leave deductions remain unchanged unless manually adjusted.

What happens when an employee takes unpaid leave mid-month?

The system calculates the deduction using the configured daily wage divisor multiplied by the number of unpaid leave days. For mid-month departures with unpaid leave, EOS calculations use the full monthly salary without adjusting for actual days worked.

Why does the Number of days field disappear when I select Working days?

When Working days is selected, the system automatically uses the working days count from the employee's work timing configuration. The custom days field is not needed because the value is derived from the work schedule.

Which calculation method should I use: Calendar days, Working days, or Custom days?

Most UAE and KSA companies use 30 custom days to align with labor law requirements. Use Calendar days if your contracts specify actual month length, or Working days if you want to exclude weekends and holidays from calculations.
